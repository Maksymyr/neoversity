# Завдання 3 — порівняння алгоритмів сортування

Емпіричне порівняння трьох алгоритмів сортування за часом виконання:
- `insertion_sort`: `O(n²)` у середньому, `O(n)` на майже впорядкованих даних;
- `merge_sort`: стабільне рекурсивне `O(n log n)`;
- `timsort`: вбудоване `sorted()` (Timsort — гібрид merge sort + insertion sort на коротких ранах, реалізація на C).

Час вимірюється `timeit.Timer(...).repeat(repeat=3, number=1)`, береться **мінімум** із трьох запусків (стандартна практика для бенчмарків — виключає шум планувальника ОС).

## Запуск

```bash
python sort_benchmark.py
```

## Тести

```bash
python test_sort_benchmark.py
```

## Висновки

1. **Insertion sort** добре поводиться лише на дуже коротких або вже впорядкованих масивах — там реалізується його `O(n)` найкращий випадок (на `sorted-100` він навіть швидший за чистий merge: 5 µs vs 58 µs). На випадкових `10 000` він уже на ~3 порядки повільніший за merge і на ~4 порядки за Timsort. На зворотньо відсортованих `10 000` — 2.4 секунди (найгірший випадок `O(n²)`). На `100 000` запуск був би ~2 хвилини, на `1 000 000` — ~3 години, що підтверджує квадратичний характер росту.

2. **Merge sort** стабільно тримає `O(n log n)` на будь-яких розподілах. Емпірика підтверджує теорію: при зростанні `n` у 10 разів час росте у ~12 разів (для `n log n` теоретичний коефіцієнт ≈ 11.7):
   - random: 13.17 мс (10k) → 160.43 мс (100k) → 1.953 с (1M) — рівно `×12.2` і `×12.2`;
   - sorted: 8.53 → 99.03 → 1142 мс — `×11.6` і `×11.5`.
   На коротких або «майже відсортованих» наборах merge гірший за insertion через накладні витрати на копіювання списків і рекурсію (на `nearly_sorted-1000`: merge 813 µs vs insertion 414 µs).

3. **Timsort** виграє на всіх досліджуваних розмірах і розподілах. На відсортованих/майже відсортованих даних він стає майже лінійним за рахунок виявлення готових «ранів»: на `sorted-1M` — 4.46 мс проти 1.14 с у чистого merge (різниця у **~256 разів!**). На випадкових даних також на 1–2 порядки швидший за чистий merge sort за рахунок:
   - реалізації на C (а не Python);
   - мінімізації порівнянь («galloping mode» при злитті ранів);
   - уникнення зайвих копіювань;
   - insertion sort на коротких підмасивах (де insertion реально швидший).

4. **Чому Timsort = merge + insertion.** Це видно прямо з таблиць:
   - на `random-100` чистий insertion (102 µs) гірший за чистий merge (78 µs), але обидва набагато гірші за Timsort (3 µs);
   - на `sorted-100` чистий insertion (5 µs) набагато кращий за merge (58 µs);
   - Timsort бере **краще з обох**: на коротких/відсортованих ділянках працює як insertion, на довгих/випадкових — як merge.

5. **Чому програмісти не пишуть свої сортування.** Поєднання двох факторів — алгоритмічних оптимізацій Timsort (insertion на коротких ранах, виявлення впорядкованих ділянок, galloping mode) і реалізації на C — робить вбудований `sorted()` практично незборимим. Власна реалізація навіть теоретично оптимального `O(n log n)` алгоритму на чистому Python *завжди* буде на 1–3 порядки повільнішою. Тому в реальному коді слід використовувати вбудовану функцію; ручна реалізація сортувань має сенс лише як навчальна вправа або у вузькоспеціалізованих кейсах.
