# algo_hw_topic_05 — Алгоритми пошуку підрядка

Емпіричне порівняння трьох класичних алгоритмів пошуку підрядка на двох текстових статтях:

- **Кнут-Морріс-Пратт (KMP)** — `O(n + m)` гарантовано, використовує prefix-функцію (LPS) для пропуску вже зіставлених символів;
- **Боєр-Мур (BM)** — `O(n·m)` у найгіршому випадку, але `O(n/m)` у типовому за рахунок евристики «поганого символу» (зрушення на основі останнього неспівпадіння);
- **Рабін-Карп (RK)** — `O(n + m)` у середньому, `O(n·m)` у найгіршому, через хешування з ковзним хешем.

Час вимірюється `timeit.Timer.repeat(repeat=5, number=1000)` — для кожної комбінації виконується 1000 викликів × 5 повторів, береться **мінімум** і ділиться на 1000 (час одного виклику).

## Підрядки для пошуку

| тип      | значення          |
| -------- | ----------------- |
| match    | `структури даних` |
| no_match | `рожевий поні`    |

`match` — підрядок, який гарантовано присутній в обох статтях (тема обох — структури даних), алгоритм знаходить перше входження. `no_match` — підрядок, якого немає в жодному тексті, алгоритм проходить весь текст і повертає `-1` (саме цей сценарій — найгірший випадок, де чітко видно різницю у складності).

## Результати

Заміри: best of 5 × 1000 викликів, середній час одного виклику.

### article_1 (12 655 символів)

| pattern  | KMP       | Boyer-Moore   | Rabin-Karp | winner      |
| -------- | --------- | ------------- | ---------- | ----------- |
| match    | 40.92 µs  | **10.75 µs**  | 129.54 µs  | Boyer-Moore |
| no_match | 992.37 µs | **271.39 µs** | 3.23 ms    | Boyer-Moore |

### article_2 (17 590 символів)

| pattern  | KMP     | Boyer-Moore   | Rabin-Karp | winner      |
| -------- | ------- | ------------- | ---------- | ----------- |
| match    | 3.28 µs | **2.19 µs**   | 6.78 µs    | Boyer-Moore |
| no_match | 1.41 ms | **384.76 µs** | 4.59 ms    | Boyer-Moore |

### Підсумок (сума по обох текстах і обох підрядках)

## Висновки

1. **Найшвидший для article_1:** Boyer-Moore (10.75 µs / 271.39 µs) — у ~3.8× швидший за KMP на `match` і у ~3.7× — на `no_match`.

2. **Найшвидший для article_2:** Boyer-Moore (2.19 µs / 384.76 µs) — у ~1.5× швидший за KMP на `match` і у ~3.7× — на `no_match`.

3. **Загальний переможець:** **Boyer-Moore** — найшвидший на всіх 4 комбінаціях (`текст × підрядок`). Сумарно у ~3.7× швидший за KMP і у ~12× швидший за Rabin-Karp.

4. **Чому Boyer-Moore виграє.** Алгоритм порівнює патерн справа наліво і при неспівпадінні використовує таблицю поганого символу: якщо в тексті побачили символ, якого взагалі немає в патерні, можна одразу зрушити вікно на всю довжину патерну. На українському тексті алфавіт великий (літери + розділові знаки + цифри), тож більшість «невипадкових» символів у тексті відсутні у короткому патерні — це дає великі зрушення. Особливо ефектно це проявляється на `no_match` підрядку: оскільки збігу немає взагалі, BM просто пропускає текст великими кроками й виконує суттєво менше порівнянь, ніж лінійні KMP/RK.

5. **Чому KMP стабільний середняк.** KMP завжди робить `O(n + m)` порівнянь — це чудова теоретична гарантія, але на практиці він порівнює символи по одному зліва направо й не вміє «перестрибувати» цілі шматки тексту, як це робить BM. Тому на коротких типових патернах він повільніший за BM приблизно на сталий множник.

6. **Чому Rabin-Karp найповільніший.** Здавалося б, RK має теоретичну `O(n + m)` складність, але в Python кожна ітерація потребує: множення великих цілих, операцій по модулю `(1 << 61) - 1`, обчислення `ord()`. Це повільніше за прості порівняння символів, які роблять KMP/BM. Перевага RK з'являється лише при пошуку **багатьох** патернів одночасно (можна порівнювати хеші) або у мовах з нативною арифметикою — але не в Python для одного патерну.

7. **Загальне правило.** Для пошуку **одного** підрядка в одному тексті — Boyer-Moore практично завжди оптимальний. Для пошуку патернів зі специфічною структурою (періодичні, з повторами) — KMP даватиме передбачувано добрі результати. Rabin-Karp виграє лише в задачах множинного пошуку (наприклад, фільтрування кількох сотень доменних імен у потоці URL).

8. **Емпіричне підтвердження теорії.** Найгірший випадок BM `O(n·m)` ми не побачили: для «природних» текстів алфавіт різноманітний, що нівелює патологію. KMP не виграв жодного забігу — підтверджується думка, що його `O(n+m)` гарантія купується ціною неможливості «стрибати» по тексту. RK у Python програв на порядок навіть на прискореній 61-бітній арифметиці.

Отже, для пошуку рідкого/відсутнього короткого підрядка у природній мові Boyer-Moore — оптимальний вибір.

## Обмеження експерименту

Висновки вище зроблено в межах конкретного сетапу — варто враховувати такі обмеження:

- **Розмір текстів невеликий** (~13K і ~17K символів). На текстах у мегабайти і гігабайти співвідношення часів може змінитися: BM зберігатиме перевагу, але різниця з KMP може зменшитись через накладні витрати на dict-lookup у bad-character таблиці.
- **Patern `match` знаходиться рано** в обох статтях (позиція 481 в article_1, позиція 10 в article_2 — практично у заголовку). Тому `match` міряє «знайти на початку», а не амортизовану вартість обходу. Якби патерн був ближче до кінця тексту, цифри `match` наблизились би до `no_match`.
- **Довжини патернів відрізняються:** `match` = 15 символів, `no_match` = 12. BM зрушує кроками до `m`, тож на коротшому патерні робить більше ітерацій. Для повністю чесного порівняння довжини мали б бути однаковими.
- **BM реалізовано лише з bad-character heuristic**, без good-suffix rule. На патернах із самоповторами (`"abab"`, `"aaaa..."`) повний BM був би теоретично кращим, але на природному тексті ця різниця невелика.
- **Тематика обох текстів схожа** (структури даних / алгоритми) — тож «існуючий» патерн `структури даних` природно з'являється в обох. На текстах різних доменів результати залежали б від конкретного патерну.
- **Усі цифри отримано на одній машині** в одному прогоні. Best-of-5 нівелює короткочасний шум, але абсолютні значення ±5–10% варіюються між запусками — для якісних висновків (хто переможець) це не впливає.
